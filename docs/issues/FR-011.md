# [FR-011] Public Website and Admissions-Ready Content

## Labels
`MUS`, `website`, `marketing`, `seo`, `domains`, `routing`

## User Story
As a prospective family, I want a polished school website with clear information and calls to action, so that I can learn about the school and contact it easily.

## Proposed Solution

### Overview
This issue is implemented in the dedicated tenant public-site app at `apps/sites`. The current `apps/www` surface remains the SchoolOS marketing site, so this FR should be read as the school-facing public website surface, not the product marketing site. The public-site runtime now supports platform-managed public subdomains and school-owned custom public domains through domain-mapping records, canonical host rules, and school-aware SEO outputs.

T23 adds the managed delivery workflow above that runtime: the platform team selects from the approved template library, collects the initial content pack, secures school approval, and handles page expansion through controlled change requests. Schools can still stay on an external website in Mode B, then later move into a managed Mode C site without re-onboarding the school record.

### Implementation Flow
1. Classify the school into Mode B (external site) or Mode C (managed site) during onboarding.
2. Resolve school context from the request hostname and its public domain-mapping record.
3. Load school branding, contact details, template key, page content, and domain readiness state at runtime.
4. Collect the initial content pack, choose an approved template, and map the content into the current template contract.
5. Redirect any non-canonical active host safely to the school’s active canonical public host.
6. Render the homepage plus essential supporting public pages from the site template contract.
7. Emit canonical metadata, structured data, robots, and sitemap output from the active school domain.
8. Keep the tenant public website isolated from the SchoolOS marketing site and from internal workspace surfaces.

### Key Considerations
- The design must feel premium and intentional, not generic.
- The site must stay reusable for later schools through one shared brand/config layer.
- Domain onboarding should be supportable through config and verification state, not route-specific hacks.
- Managed delivery is platform-led, not a self-serve page builder.
- Admissions should stay informational; no workflow automation is required for this FR.
- External-site schools must remain supported even when they do not adopt a platform-built public site.

## Acceptance Criteria

- [x] The public site supports school-specific branding and copy.
- [x] The homepage is implemented and mobile-first.
- [x] Essential supporting pages are implemented from the sitemap: About, Academics, Admissions, Fees, Visit, and Contact.
- [x] Public CTAs and contact intent flows are present.
- [x] The site supports platform-managed public subdomains through domain records rather than flat hostname lists.
- [x] The site supports school-owned custom public domains with verification token, readiness, and SSL state.
- [x] Non-canonical active hosts redirect safely to the canonical active school domain.
- [x] Metadata, structured data, robots, and sitemap output resolve from the active canonical school domain.
- [x] Unknown, inactive, and not-yet-active hosts fail safely.
- [x] The implementation remains isolated to the dedicated public `sites` surface and does not repurpose the SchoolOS marketing site.
- [x] The managed delivery policy defines platform-led template selection and content approval instead of a full self-serve builder.
- [x] Schools that keep their own public website still have a supported Mode B path plus a documented migration path to Mode C.
