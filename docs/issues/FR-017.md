# [FR-017] Security, Permissions, and Auditability

## Labels
`MUS`, `security`, `audit`

## User Story
As a school owner, I want strong authorization, rate limiting, audit logging, and safe support access, so that sensitive student and payment data stays protected.

## Proposed Solution

### Overview
Layer school-aware authorization, read-only support inspection, audit logs, and rate controls across auth, academic, payment, and AI workflows.

### Implementation Flow
1. Define permission matrix by role and school context.
2. Add audit logging for sensitive actions.
3. Apply rate limits to high-risk endpoints and actions.

### Key Considerations
- Student and payment data are high sensitivity.
- Support access must not become hidden impersonation.

## Acceptance Criteria

- [ ] Sensitive actions are permission-checked and logged.
- [ ] Admin support views remain read-only.
- [ ] High-risk routes and actions are rate-limited.

## Current Alignment Snapshot

What is already aligned:

- platform admin and school admin permissions are now separated
- tenant boundaries are explicit in current Convex authorization paths
- school and platform auth no longer rely on stuffing app roles into Better Auth user fields

What is still missing:

- audit logs for sensitive platform and school actions
- explicit support-access tooling
- read-only support mode
- broader rate limiting on high-risk flows

Important future note:

- support access should be implemented as explicit, auditable support tooling
- do not solve support by making the platform owner a permanent admin inside every school
