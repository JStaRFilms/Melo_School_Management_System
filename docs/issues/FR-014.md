# [FR-014] School-Scoped Paystack Setup and Front-Desk Payments

## Labels
`MUS`, `payments`, `billing`, `admin-ui`

## User Story
As a school admin or front-desk bursar, I want to configure Paystack-backed online payments for the school and generate a payment link for a specific invoice, so that families can pay the school safely without using the parent portal.

## Proposed Solution

### Overview
Add a school-scoped Paystack setup workflow in the billing admin surface, then let staff open an invoice and generate a shareable payment link for a front-desk handoff. The payment link should be QR-ready or copyable as a URL, and webhook reconciliation must update the correct school invoice safely.

### Implementation Flow
1. Add school-level billing settings for invoice prefix, currency, due days, and online-payment toggles.
2. Use a school-owned Paystack merchant model where school and invoice metadata are embedded in the checkout payload and the active merchant secret is resolved per school and mode.
3. Generate invoice payment links from the admin billing workflow.
4. Display the payment URL in a handoff panel that can be copied or opened on a cashier device.
5. Send the payer back to a public return page that verifies the Paystack reference programmatically on their device.
6. Reconcile successful Paystack webhooks back to the correct school invoice as a background fallback.
7. Reject or ignore duplicate, invalid, or malformed events defensively.

### Key Considerations
- Paystack credentials must stay school-scoped and secret material must never leak through normal UI reads.
- School settings must never leak or mix with future platform SaaS billing.
- Payment links must stay invoice-specific and school-specific.
- Webhook processing must be idempotent so repeated deliveries do not double-count money.

## Acceptance Criteria

- [x] A school admin can enable and configure Paystack-backed online payments for school invoices.
- [x] From an invoice, the admin can generate a payment link for the parent.
- [x] The admin can display or copy a QR/payment link for the parent to complete on their own device.
- [x] Successful Paystack callbacks update the right invoice and payment records safely.
- [x] Failed, duplicate, or invalid webhook events are handled defensively.
- [x] The implementation is safe for a multi-school deployment model and does not conflate school money with platform SaaS billing.

## Alignment Note

This issue is strictly about **school billing**.

It should not include:

- parent self-serve portal billing
- platform SaaS subscription billing
- per-school merchant orchestration beyond the MVP direct-to-school model

## Implementation Note

The implementation now uses a school-owned Paystack merchant flow:

- school billing settings configure invoice prefix, currency, due days, online-payment toggles, and the active Paystack mode (`test` or `live`)
- school admins can save and validate per-mode Paystack credentials from the billing workspace with masked state only
- the admin can generate a payment link from an invoice for front-desk handoff using the correct school-owned merchant secret
- the default callback target is now a public Paystack return page, so parents do not get redirected into the admin workspace
- the public return page verifies the payment reference programmatically without requiring an admin confirmation click
- durable payment-attempt records now preserve pending references and provider-mode context for later reconciliation when the payer switches devices or skips the return page
- the admin workspace can recheck pending Paystack references without turning manual confirmation into the primary workflow
- webhook verification now resolves the correct school and merchant mode before verifying with that school-specific secret and still guards against duplicates as a background fallback
- the school billing workspace remains separate from future platform SaaS billing
