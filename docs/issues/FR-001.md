# [FR-001] Tenant-Aware School Setup

## Labels
`MUS`, `platform`, `tenancy`

## User Story
As a platform owner, I want each school to have independent branding, settings, and academic configuration, so that the same codebase can serve multiple schools.

## Proposed Solution

### Overview
Create a tenant model centered on `schoolId` and `schoolSlug`, with branding, feature flags, academic defaults, and school-scoped settings available to all apps.

### Implementation Flow
1. Define school and branding schemas.
2. Add school-aware loaders and context helpers.
3. Use school slug or mapped domain to resolve tenant context.

### Key Considerations
- School boundaries must be explicit in backend and UI code.
- Branding must be portable across public and authenticated apps.

## Acceptance Criteria

- [ ] School records include branding, slug, and configuration fields.
- [x] Public and authenticated apps can resolve active school context for the current school and platform-provisioning flows.
- [x] Cross-school data leakage is prevented by default in the current multi-tenant backend patterns.

## Current Alignment Snapshot

What is already aligned:

- school records and slug-based tenant provisioning exist
- platform super admin can create schools and assign first school admins
- school-scoped authorization is enforced through `schoolId`
- platform authorization is separated through `platformAdmins`

What is still missing for full FR-001 completion:

- richer school branding/default-brand pack setup
- domain-to-school resolution for white-label public sites
- broader school configuration fields beyond the current provisioning slice
- authenticated workspace branding for `admin`, `teacher`, and `portal` based on resolved school context

Recommended next follow-on when revisiting this issue:

- connect school branding + metadata to authenticated surfaces and public domain mapping
- preserve the separation between authenticated role hosts and school identity, so school theming does not depend on school-specific subdomains
