# Public Site and SEO Foundation

## Goal

Provide a polished public marketing site for SchoolOS in `apps/www` that explains the product, stays mobile-first, and leaves school public websites for the later T20-T23 public-web work.

## Scope Boundary

This feature covers the public marketing surface only:

- homepage
- overview page
- modules page
- packages page
- commercials page
- demo page
- contact page
- metadata, robots, sitemap, and structured data

It does **not** include:

- internal workspace routing or auth
- admissions workflow automation
- school public website runtime
- enquiry form backends
- CRM handoff logic
- marketing-media automation
- admin/teacher/portal changes

## Data Flow

1. The app layout injects a product theme through CSS variables so the surface can be rebranded later.
2. Shared product copy, navigation, metadata helpers, and page content arrays live in `apps/www/lib/site.ts`.
3. The homepage renders structured data for the SchoolOS product and provides the core conversion path for prospective buyers.
4. Supporting pages expand the product story across overview, modules, packages, commercials, demo, and contact paths.
5. `robots.ts` and `sitemap.ts` expose a clean public crawl surface with canonical URLs.

## Implemented Outcome

- `apps/www` now renders as the SchoolOS marketing site.
- The homepage communicates the product story, trust signals, module groups, and commercial entry points.
- Supporting pages cover overview, modules, packages, commercials, demo, and contact content.
- Metadata is in place for titles, descriptions, canonical URLs, open graph data, and Twitter cards.
- `robots.txt`, `sitemap.xml`, and a simple favicon are generated by Next.js.
- The content model stays reusable because product branding, contact details, and copy live in one shared configuration file.

## Regression Checks

- The public site must remain separate from admin, teacher, portal, and internal platform surfaces.
- Navigation must stay mobile-friendly and easy to scan.
- Copy should stay product-facing rather than school-admissions-facing.
- Public CTAs must not expose `/schools` or internal workspace URLs.
- Future school public website work should stay in the T20-T23 backlog.
